# Repository Guidelines

## Project Structure & Module Organization
Entrance Tools is a Node/Express monolith. `server.js` owns HTTP/WebSocket routing plus auth/token signing, SSH, SFTP, local shell, flash/debug, VNC, Docker stats, known-host tracking, and private-network allowlist wiring, while `local-shell.js` provides the cross-platform shell layer (Linux/macOS via `script`, Windows via direct shell spawn), `flash-debug.js` wraps OpenOCD/pyOCD/probe-rs plus optional privilege elevation, `vnc.js` handles the VNC proxy, and `vnc-client.js` handles browser plumbing. The editable WebUI now lives under `webui-src/`: `index.template.html` composes HTML partials from `webui-src/partials/`, `webui-src/styles/app.css` holds the stylesheet, and `webui-src/scripts/app.js` holds the application logic. `scripts/build-webui.js` generates `public/index.html` plus `public/assets/app.css` and `public/assets/app.js`; treat `public/` as build output and change `webui-src/` instead. `start.sh` is the preferred local launcher: it pins runtime data to `./.data`, creates `./.data/auth_secret` on first boot when `.data` does not yet exist, optionally accepts `--port=4000`, exports `ENTRANCE_DATA_DIR` and `AUTH_SECRET`, and then runs `npm start`. `start_nocors.sh` wraps `start.sh` but exports `ENTRANCE_CORS_DISABLE=1` first for LAN, reverse-proxy, or tunnel browser access. Container deployment assets live in `Dockerfile`, `compose.yml`, and `.dockerignore`. Runtime state lives in `users.json`, `userdata/*.json`, `known_hosts.json`, `private-networks.json`, `LOGIN_KEEP`, and `.ssh_password_key` under `ENTRANCE_DATA_DIR` (default repo root); never commit real user data or generated secrets. Do not upload or commit `.data/`, any `ENTRANCE_DATA_DIR` contents, or any other user/runtime data directories. Temporary payloads land under `uploads/` and dependencies live in `node_modules/`.

Documentation is English-first: keep the root `README.md` as the default English README, keep the Simplified Chinese translation in `doc/README_CN.md`, use `doc/screenshot.png` from the root README, and use `screenshot_cn.png` inside `doc/README_CN.md`. When updating repository guidance, keep `AGENTS.md` and `CLAUDE.md` in English. Preserve each document's existing language when editing it; do not mix languages inside a file unless that file already does so intentionally. When modifying a Markdown document, keep the document in its existing language. When `README.md` changes, update every corresponding `README_XX.md` translation document in the repository in the same pass; in this repository that means `doc/README_CN.md` must stay aligned with `README.md`.

## Collaboration Preference
When the user provides a numbered list of requirements such as `1.` and `2.`, first reply with the implementation plan and wait for approval before making changes.

The frontend uses modular JavaScript objects (`State`, `Storage`, `Theme`, `Settings`, `I18n`, `Toast`, `Terminal_`, `FlashDebug`, `SFTP`, `Hosts`, `UI`) with CSS variables for theming. Color schemes are applied via a `data-color-scheme` attribute on `<html>` (valid values: `default`, `sakura`, `ocean`, `forest`, `twilight`, `amber`) and work orthogonally with the `data-theme` light/dark toggle. The SPA now also supports UI i18n with English as the default language and a Simplified Chinese translation option. The `Settings` module provides an in-app password change form (calls `PUT /api/users/:username/password`), a login-keepalive card with presets (`7d`, `14d`, `1m`, `never`) plus custom duration input, an admin-only private-network allowlist card rendered below the keepalive/password area, a color scheme picker, and a separate language card rendered below the color scheme card; language selection is persisted in `localStorage`. In desktop wrapper no-login mode, the password form is hidden and replaced with a notice.
The same auth overlay now doubles as a staged startup splash: when restoring a saved login and desktop no-login is not active, show the Material You wave animation with `public/logo.png` clipped to a rounded rectangle and keep a minimum 3-second progress sequence while the dashboard shell renders first and backend-backed modules initialize afterward. The loading/splash markup lives in `webui-src/partials/auth-overlay.html`, the related styling lives in `webui-src/styles/app.css`, and the runtime loading/boot sequencing lives in `webui-src/scripts/app.js` (`showLoading`, `updateLoadingProgress`, `startDashboardBoot`). In secure desktop deployments, the Electron wrapper should render local frontend assets, keep the admin token in the main process, call `POST /api/auth/desktop/bootstrap` with `X-Entrance-Desktop-Secret`, and let the backend run with `ENTRANCE_DESKTOP_API_ONLY=1` so browsers cannot fetch the no-login token.

The flash/debug view now uses a shared autocomplete flow for tool-specific target inputs. OpenOCD loads searchable `target/*.cfg` and `interface/*.cfg` catalogs from its discovered scripts directory, pyOCD loads searchable target names from `pyocd list --targets --no-header`, and probe-rs loads searchable chip names from `probe-rs chip list`. All of these feed the same frontend autocomplete renderer in `FlashDebug`, which supports prefix, substring, and lightweight fuzzy matching while still allowing free-form manual input.
The same shared path also drives Flash & Debug placeholders, helper hints, candidate counters, and fallback probe/catalog notices, and those runtime strings are expected to follow the active UI language for all three tool families.

The SSH view includes three collapsible monitoring panels below the terminal: System Stats (CPU/Memory/Disk I/O line chart, 1s interval), TOP process list (2s interval), and Docker Stats (container resource ring charts for CPU/MEM/NET IO/BLOCK IO, 3s interval with total/single-container view modes). All panels share the same polling pattern: server-side `collectXxx()` via `sshClient.exec()` → `setInterval` → WebSocket push → client `handleXxx()` → DOM render.

The flash/debug view is admin-only and runs local OpenOCD, pyOCD, or probe-rs commands through `flash-debug.js`. The UI can optionally request elevated privileges before launch: Linux prefers `pkexec` and falls back to `sudo` with `zenity`/`kdialog`, macOS uses `sudo` with `osascript`, and Windows prefers `gsudo` and otherwise uses `sudo`.

## Build, Test, and Development Commands
`npm install` sets up Express, ws, ssh2, multer, archiver, and argon2.
`npm run build:webui` regenerates `public/index.html` and `public/assets/*` from `webui-src/`.
`./start.sh` is the preferred local startup path; it manages `./.data`, exports `ENTRANCE_DATA_DIR` plus `AUTH_SECRET`, and then runs `npm start`. Pass `--port=4000` to make it call `npm start -- --port 4000`.
`./start_nocors.sh` behaves the same way, but exports `ENTRANCE_CORS_DISABLE=1` before delegating to `./start.sh`.
`npm start` (and its alias `npm run dev`) runs `npm run build:webui` and then executes `server.js` on port 3000; it requires `AUTH_SECRET` to already be exported. Override with `./start.sh --port=4000`, `./start_nocors.sh --port=4000`, `PORT=4000 npm start`, or `npm start -- --port 4000`.
`docker build -t entrance-tools .` builds the container image, while `docker compose up -d --build` uses `compose.yml` and persists data under `./data`; `PORT=4000 docker compose up -d --build` remaps both the container listener and published port. Podman users can substitute `docker` with `podman` for equivalent build/run flows.
`NODE_ENV=development ./start.sh` enables verbose logging with the default local bootstrap flow; `NODE_ENV=development npm start` also works if `AUTH_SECRET` is already exported. Pair with a reverse proxy when deploying publicly.

## Environment Variables
- `AUTH_SECRET` is required and signs auth tokens; `start.sh` bootstraps it from `./.data/auth_secret` for the default local workflow.
- `SSH_PASSWORD_KEY` encrypts stored SSH credentials and private-network allowlists; if omitted, the server writes a generated key to `.ssh_password_key` under `ENTRANCE_DATA_DIR`.
- `PORT` controls the HTTP listening port; the same value can also be overridden at startup with `--port` / `-p`.
- `ENTRANCE_HOST` overrides the bind address; web mode defaults to `0.0.0.0`, while desktop API-only mode defaults to `127.0.0.1`.
- `ENTRANCE_DATA_DIR` relocates runtime state such as `users.json`, `userdata/`, `known_hosts.json`, `private-networks.json`, `LOGIN_KEEP`, and `.ssh_password_key`.
- Security and targeting knobs include `AUTH_TOKEN_TTL`, `LOGIN_WINDOW_MS`, `LOGIN_MAX_ATTEMPTS`, `STRICT_HOST_KEY_CHECKING`, `ALLOWED_TARGETS`, `ALLOW_PRIVATE_NETWORKS`, `ENTRANCE_CORS_DISABLE`, `ENTRANCE_DESKTOP_NOLOGIN`, `ENTRANCE_DESKTOP_API_ONLY`, `ENTRANCE_DESKTOP_ALLOWED_ORIGIN`, and `ENTRANCE_DESKTOP_BOOTSTRAP_SECRET`. `AUTH_TOKEN_TTL` now defaults to 7 days for password logins unless the user overrides it from Settings.

## Coding Style & Naming Conventions
Backend files use four-space indentation, semicolons, and CommonJS `require` + `module.exports`. Prefer `const`, destructure imports, and group helpers in plain objects (see `UserManager`). Filenames stay kebab-case, whereas functions/variables use lower camelCase. Front-end additions should blend into the Fluent Design theme defined by `webui-src/index.template.html`, `webui-src/styles/app.css`, and `webui-src/scripts/app.js`; edit source files and regenerate `public/`, instead of hand-editing generated assets. Anytime you add dependencies, re-run `npm install` so `package-lock.json` remains consistent.

## Testing Guidelines
Automated tests are not yet wired in, so validate changes manually: run `./start.sh`, `./start_nocors.sh`, or `npm start` with `AUTH_SECRET` already exported as appropriate, or use the relevant container flow. Log in with a disposable or the default `admin/admin` account on a safe machine, and exercise each updated surface (SSH terminal, local shell on the target OS, flash/debug including optional elevation flow when touched, VNC, Serial, SFTP, private-network management when touched, and the staged startup splash / session-restore flow when auth or boot behavior changes). When desktop auth/bootstrap behavior changes, also verify `ENTRANCE_DESKTOP_API_ONLY=1` with and without the bootstrap secret so `/api/auth/nologin` no longer leaks an admin token. Capture the commands you issued and browsers/OSes used, then paste that checklist into the PR. For new utility modules, add lightweight `node scripts/my-check.js` assertions or document how reviewers can replicate the scenario.

## Commit & Pull Request Guidelines
Commits follow Conventional Commits (`feat:`, `fix:`, `docs:`, etc.); keep patches focused and avoid bundling unrelated fixes. PRs should explain the problem, the approach, and the validation evidence (screenshots, terminal recordings, or log snippets). Highlight any migrations touching `users.json`, `userdata/`, `known_hosts.json`, `private-networks.json`, `LOGIN_KEEP`, `.ssh_password_key`, or container deployment behavior so deployers can back up state. When adding new SSH monitoring panels, follow the existing collectXxx/startXxx/stopXxx/refreshXxx pattern in both `server.js` and the `Terminal_` object in `webui-src/scripts/app.js`. When changing flash/debug behavior, update both `flash-debug.js` and the `FlashDebug` object in `webui-src/scripts/app.js`, including any privilege-elevation messaging. If you touch target-search behavior, keep the shared autocomplete path aligned across OpenOCD config catalogs, pyOCD target catalogs, and probe-rs chip catalogs rather than reintroducing tool-specific UI implementations.
If you touch Flash & Debug helper copy, placeholders, or autocomplete status text, keep the shared I18n path aligned as well so English/Chinese stay consistent across OpenOCD, pyOCD, and probe-rs.
If you touch login bootstrap, remember the staged startup constraints: browser restores should use the web splash with a minimum 3-second fake progress path, while secure desktop no-login should be wrapper-owned, API-only, and bootstrapped through `POST /api/auth/desktop/bootstrap`.

## Security & Configuration Tips
Passwords in `users.json` are Argon2id hashes, not plaintext. Stored SSH credentials and private-network allowlists are AES-256-GCM encrypted with `SSH_PASSWORD_KEY` (or the generated `.ssh_password_key`), so treat key rotation as a migration. `LOGIN_KEEP` stores the last successful password-login Unix timestamp encrypted with AES-256-GCM using a key derived from `AUTH_SECRET`, and saved browser sessions should only remain reusable while `AUTH_SECRET` stays stable. Prefer environment variables for `AUTH_SECRET`, `SSH_PASSWORD_KEY`, `ENTRANCE_DATA_DIR`, host allowlisting, and login throttling. `ENTRANCE_CORS_DISABLE=1` intentionally widens browser access and should only be used when you need LAN, reverse-proxy, or tunnel origins; `start_nocors.sh` exists as a convenience wrapper for that mode. In desktop no-login deployments, do not expose the web UI over localhost and do not rely on renderer-visible tokens; use `ENTRANCE_DESKTOP_API_ONLY=1`, loopback binding, and the bootstrap secret flow instead of widening CORS. Sanitize any hostnames or file paths taken from the UI, keep private-network rules narrow, and develop using a non-privileged system account. Flash/debug elevation depends on trusted local executables (`pkexec`, `sudo`, `zenity`, `kdialog`, `osascript`, `gsudo`) and should remain admin-only. Never upload or commit `.data/`, `userdata/`, generated runtime JSON, or any other user data snapshots; remove throwaway files from `uploads/` and runtime data directories before requesting review.
